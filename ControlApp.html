ControlApp

Check out the ControlApp Repository on GitHub!

ControlApp is an application that begun with a very simple purpose. My clients wanted a way to change which Tar Heel Reader 'Bookshelf' (a url to a specific THR favorite's page) the user was redirected to and in turn, limit their access to only a certain selection of books. This originally was hard coded into the EyeReader code. There were two options: the obvious solution was simply to make another version of the software that would be switched out when needed. But, this in itself would basically require the user to 'redownload' the software and I didn't want to make the user go through a possibly arduous situation twice. The program can take a decent amount of time to sync in dropbox when one has very slow internet speeds.

My other option was to get a bit creative.

I figured all I had to do was make a tiny little text file, called "Bookshelf.txt", whose sole purpose was just to either have 'A' or 'B' which told the program which of two bookshelves to redirect the user's browser to. But this in itself caused a bit of an issue. I didn't want them having to hunt through the program files to change this. And what if this file read something other than A or B? What then?

The ControlApp was a simple user interface designed to allow research administrators like my clients to alter the program to their needs. And it didn't stop at being just a simple tool to change one text file. It lead me to series of questions that I began to have about the future of their research... a future that I might not always be around (as a research assistant for a semester) to fix. How could I plan for future troubleshooting? How could I automate and improve upon this idea of data driven research through Dropbox?

The second thing I added was the user-bookshelf list. Its simple enough, as all it really does it read the 'Bookshelf.txt' of each user to display which user corresponds to what bookshelf. But then I also had a thought... my clients had mentioned before that they waited a specific period of time before they switched a user to the next bookshelf. Would it not be useful then to know when the last time they had switched the user's bookshelf? And thus, the extra text file 'Bookshelf_Date.txt' was born; updated every time the administrator changed which bookshelf the user was set to.

I soon became enamored with this idea of a interface to control all instances of the program in use. Wasn't that part of the beauty of Dropbox? Having access to all of your user's applications in order to gather data? But it could be so much more than that. This was the idea that was put into practice.

And thus, the third feature, the 'Create New User' was made. Why go through the tedious process of copying another user's folder, removing their data files, and then renaming it? Instead I chose to automate this process with a little help from shutil a python module designed for high level file operations. All we do is keep an internal copy of a 'blank' version of the program and then allow the administrator to set the user's name (folder name) and bookshelf upon creation.

The fourth feature also takes advantage of shutil to allow the administrator to update and or 'reinstall' a user's program. I found this to be particularly necessary in time to come, because the fault of the Dropbox distribution design is that the program files aren't hidden away somewhere deep within the system where normal computer users will never touch. There are instances where the program might be corrupted one way or another. Therefore, its important to have a way to reinstall them with a fresh version of the software without losing all of the precious data collected.

The final feature is the most important of all. As seen above, it allows the administrator to create either CSVs, Line Graphs, or Picture Graphs of the data collected from the users (to see more about these, check out Graphs and Data Analysis). Clicking any of the buttons above opens up a file explorer for the administrator to select files from a user.

All in all, I think even the general idea of the ControlApp would be an important addition to any data driven design featuring dropbox. And it can be as simple as a few looping easygui popup windows as mine is.
